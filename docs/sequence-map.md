# 시퀀스 맵

## 이 레포 안의 성장 흐름

### 02. 영속성 저장과 계층 분리

- 메모리 CRUD를 MySQL 기반 CRUD로 바꿉니다.
- `Controller -> Service -> Repository -> DB` 흐름을 익힙니다.

### 03. 안전한 요청 처리

- 요청 DTO와 응답 DTO를 나눕니다.
- Validation과 전역 예외 처리로 실패 응답을 통일합니다.

### 04. 인증과 JWT

- 회원가입과 로그인 흐름을 만듭니다.
- 로그인 이후 토큰으로 현재 요청 사용자를 구분합니다.

### 05. OAuth2 + SMTP + Account Recovery

05는 하나의 시퀀스 안에서 세 단계로 나눕니다.

- 05-A: Google OAuth2 redirect 후 provider, providerId, email을 읽습니다.
- 05-A: LOCAL 사용자와 OAUTH 사용자를 구분하고 자체 JWT 또는 사용자 정보 흐름으로 연결합니다.
- 05-B: `RecoveryMailSender`와 `SmtpRecoveryMailSender`로 메일 발송 책임을 분리합니다.
- 05-B: SMTP host, port, username, password는 환경변수 기반으로 설정합니다.
- 05-C: 복구 요청, 토큰 생성, 토큰 만료, 비밀번호 재설정 흐름을 작게 설계합니다.
- 05-C: 존재하지 않는 email, 만료 토큰, 재사용 토큰 같은 실패 케이스를 확인합니다.

### 06. 테스트와 검증

- `PostService`와 `AuthService`를 Service 단위 테스트로 검증합니다.
- fixture와 mock으로 테스트 입력과 의존성을 준비합니다.
- 정상 케이스와 실패 케이스를 나눠서 확인합니다.

## 왜 이 순서인가

인증은 단독으로 떠 있는 기능이 아니라,
이미 만들어 둔 저장 구조와 요청 검증 위에 얹혀야 자연스럽습니다.

그래서 이 레포는 아래 순서를 고정합니다.

1. 먼저 저장 구조를 만든다.
2. 그 다음 요청을 안전하게 만든다.
3. 그 위에 로그인과 JWT를 붙인다.
4. 그 다음 외부 로그인, 메일 발송, 계정 복구를 작은 단계로 붙인다.
5. 그 다음 이미 만든 기능을 테스트로 다시 검증한다.

이 순서를 따르면 저장, 요청 검증, 인증, 외부 연동, 테스트가 어떤 문제를 이어서 해결하는지 단계별로 볼 수 있습니다.
